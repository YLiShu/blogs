    控制台输出顺序正确的是（）
    A. 25341 B. 23541 C. 52341 D. 51234

    setTimeout(function () {
    console.log(1);
}, 0);

new Promise(function executor(resolve) {  
    console.log(2);  
    for (var i = 0; i < 10000; i++) {  
        i == 9999 && resolve();
     }
    console.log(3); 
}).then(function () {
    console.log(4); 
});

console.log(5);

    答案：23541。 解析：这是因为 Javascript 中存在事件循环，它会处理不同的任务队列，具体来说，以下是发生的情况：

    1. 首先，Promise 的执行是立即的，所以 2 和 3 会被立即输出。
    2. 然后，Promise 中的 for 循环会运行，但由于 Promise 是异步的，所以不会阻塞主线程。5 会被输出。
    3. 等待主线程执行完毕后，Promise 的 then 方法会注册一个微任务。微任务会在主任务队列的所有任务完成后立即执行，所以 4 会在 5 之后输出。
    4. 最后，setTimeout 设置的回调函数会在主任务队列中注册一个宏任务，但由于传入的延迟为 0，所以它将在当前任务队列的任务执行完之后立即执行，因此 1 会在 4 之后输出。

    以下 Js 程序的输出是什么？ A. Value is Something B. Value is Nothing C. NaN D. 以上都不对

    val = 'smtg';
console.log ( 'Value is ' + ( val  ===  'smtg' )  ?  'Something' : 'Nothing' );

    答案：这段 Javascript 程序的输出是 Something 解析：这是因为 Javascript 中有运算符优先级的概念，三元条件 ? : 的优先级比 + 高。因此，在这段代码中，表达式被解析为：

    ('Value is' + (val === 'smtg')) ? 'Something' : 'Nothing'

    由于 val 等于 smtg，所以 (val === 'smtg') 的结果为 true，因此表达式会被解释为：

    ('Value is' + true) ? 'Something' : 'Nothing'

    HTTP 协议说法正确的是？ A. HTTP是一个无状态的面向连接的协议，因此不能保持TCP连接 B. 通过开启Keep-Alive，可以保持连接特性 C. HTTP默认使用的是UDP协议（无连接） D. HTTP的管道机制（pipelining）可以确保浏览器按照顺序依次发出多个请求

    答案：通过开启 Keep-Alive，可以保持连接特性。 解析：HTTP 是一种基于 TCP 的协议，它本身是无状态的面向连接的，每个请求-响应周期都是相互独立的，不会保持持久连接。然而，通过开启 Keep-Alive（持久连接），可以在单个 TCP 连接上执行多个 HTTP 请求和响应，而不必为每个请求都建立和关闭连接，从而提高性能。 HTTP 是一个无状态的面向连接的协议，但可通过 Keep-Alive 保持连接特性，故A错 HTTP 默认使用的式 TCP 协议，而不是 UDP，而且它是面向连接的。 HTTP 的管道机制允许浏览器并行发送多个请求，而不一定按顺序。但是 HTTP 管道并没有得到广泛的支持，因此在实际中很少使用。

    以下说法正确的是 A. 同步ajax请求不会引起浏览器阻塞 B. 浏览器对同一域名的请求数量没有限制 C. url 传参没有长度限制 D. ftp请求不会自动携带cookie

    答案：FTP 请求不会自动携带 cookie 解析：

    1. 同步 AJAX 请求会引起浏览器阻塞，同步请求会阻塞浏览器的主线程，直到请求完成，这会导致用户页面的冻结，因此通常不建议在主线程中执行同步请求。
    2. 浏览器对同一域名的请求数量有限制，通常为同时允许的并行请求数。这个限制可以因浏览器类型和配置而有所不同，通常在 4 到 8 个请求之间。这是为了避免对同一域名发出过多的请求，从而防止过度占用服务器资源。
    3. URL 传参的长度有限制，这个限制通常由浏览器和服务器的配置来确定。常见的浏览器限制是 2083 字节，而服务器的限制可以更高一些。如果 URL 参数长度超过了浏览器或服务器的限制，可能会导致截断或不完整的请求。
    4. FTP（文件传输协议）是一个不同于 HTTP 的协议，Cookie 是基于 HTTP 协议的，因此在 FTP 请求中不会自动携带 Cookie，如果需要在 FTP 请求中使用身份验证信息，通常需要在 FTP 客户端中显示提供这些信息。

    以下说法错误的是 A. 浏览器的安全都是以同源为基础，如果缺少了同源策略，则浏览器的正常功能可能都会受到影响 B. HttpOnly解决的是XSS攻击后的Cookie劫持攻击 C. 存储型XSS会把用户输入的数据“存储”在客户端 D. XSS攻击，通常指黑客通过“HTML注入”篡改可网页，插入了恶意的脚本，从而在用户浏览网页时，控制用户浏览器的一种攻击

    解析：

    1. 浏览器的安全性基于同源策略，如果同源策略不得体，浏览器的正常功能可能会受到影响。同源策略是一种安全措施，它限制了来自不同源的网页间的交互，以防止恶意的跨站点请求和数据泄漏。
    2. HttpOnly 是一个安全标志，它通过将 Cookie 标记为 HttpOnly，使得 Javascript 脚本无法访问该 Cookie。这有助于防止 XSS 攻击后的 Cookie 劫持，因为攻击者无法在受感染的网站上运行 Javascript 以窃取这些 Cookie。
    3. 存储型 XSS(Stored XSS) 攻击并不是将用户输入的数据存储在客户端。它是指攻击者在受害者浏览器上存储恶意脚本，然后当用户访问包含这些恶意脚本的网页时，这些脚本会在用户的浏览器上执行，从而使攻击者能够窃取用户数据或执行恶意操作。
    4. XSS 攻击（跨站脚本攻击）是一种攻击方式，黑客通过注入恶意脚本到网页中，使用户在浏览网页时执行这些脚本，从而使攻击者能够控制浏览器或窃取用户数据。这是一种常见的 Web 安全威胁。

    段页式管理每取一数据，要访问（ ）次内存。

    答案：3次 解析：段页式管理是一种存储管理方式，它将逻辑地址分为段号和页号两部分，然后通过段表和页表将逻辑地址转换为物理地址。段页式管理每取一次数据，要访问 3 次内存。 访问内存的过程如下：

    1. 第一次是由段表地址寄存器得段表始址后访问段表，由此取出对应段的页表在内存中的地址。
    2. 第二次则是访问页表得到要访问的物理地址。
    3. 第三次才能访问真正要访问的物理单元。 补充：
    4. 页式存储（2次）
    5. 页式存储是一种存储管理方式，它将逻辑地址分为页号和页内偏移两部分，然后如果页表将逻辑地址转换为物理地址。页式存储每次从主存中取一次数据需要访问 2 次内存。
    6. 第一次访问内存中的页表，利用逻辑地址中的页号查找到页帧号，与逻辑地址中的页偏移拼接形成物理地址。
    7. 第二次，得到物理地址后，再一次访问内存，存取指令或者数据。
    8. 段式存储（2次）
    9. 段式存储是一种存储管理方式，它将逻辑地址分为段号和段内地址两部分，然后通过段表将逻辑地址转换为物理地址。段式存储每次从主存中取一次数据需要访问 2 次内存。
    10. 第一次是由段表地址寄存器得到段表始址后访问段表，从中取出对应段的基址和段长，判断是否越界
    11. 第二次是将基址和段内地址相加得到物理地址，然后访问主存中的数据或指令。

    2. 浏览器渲染过程会忽略 display:none 的 dom 元素的子元素 解析：
    3. 重绘是指当元素的外观发生改变，但不影响布局时，浏览器重新绘制元素的过程，例如改变颜色或背景等。
    4. 重排是指当元素的尺寸、位置或内容发生改变，影响了布局时，浏览器重新计算元素的几何属性并重新布局的过程，例如改变宽高或添加删除节点等。

    以下说法正确的是 A. Animal.prototype.__proto__ == Function.prototype B. Animal.prototype.__proto__ === Object.prototype C. Object.__proto__ === Function.prototype D. Function.__proto__== Object.prototype

    function Animal(name) {
	this.name = name;
}

Animal.prototype.data = {
	height: 30
}

let animal1 = new Animal('猴子');
let animal2 = new Animal('猴子');

    答案： Animal.prototype.__proto__ === Object.prototype Object.__proto__ === Function.prototype 解析：

    1. 在 Javascript 中，每个函数都有一个 prototype 属性，这个属性是一个对象，它的 __proto__ 属性指向了 Object.prototype，而不是 Function.prototype
    2. 函数的 prototype 对象的 __proto__ 指向了Object.prototype
    3. 在 JavaScript 中，Object 是一个函数，因此它的 __proto__ 属性指向 Function.prototype
    4. 在 Javascript 中，Function 是一个函数，因此它的 __proto__ 属性指向了 Function.prototype，而不是 Object.prototype

    prototype 和 __proto__ 的区别：

    1. 定义对象的方式：prototype 是函数才有的属性，它指向函数的原型对象，而 __proto__ 是所有对象都有的属性，它指向该对象的原型
    2. 对于函数 Foo，Foo.prototype 是其原型对象
    3. 对于实例 foo = new Foo()，foo.__proto__ 指向 Foo.prototype
    4. 作用：prototype 的主要作用是实现 Javascript 的继承机制。当我们使用 new 关键字创建一个对象时，这个对象的 __proto__ 属性就会指向构造函数的 prototype 属性。这样，对象就可以访问到 prototype 中定义的属性和方法
    5. 指向的对象：prototype 属性指向的是一个原型对象，这个原型对象上定义了一系列属性和方法，这些属性和方法可以被所有的实例共享。而 __proto__ 属性是一个访问器属性（getter/setter），它暴露了通过它访问到的对象内部的 [[Prototype]]（隐式原型）属性。
    6. 通过 Foo.prototype 可以向所有 Foo 实例添加共享的属性和方法。通过 foo.__proto__ 无法直接修改，但可以在实例创建时修改它指向的原型，实现继承效果
    7. Object.prototype 是最顶层的原型对象，所有对象的 __proto__ 属性都最终指向它。
